# v1.6.5 benchmark scenario — 12-turn refactor

Fixed, reproducible scenario used to measure the per-session resident-prompt
cost before and after v1.6.5's slimming changes. Run with `npx ccusage --json`
after each turn and record the cumulative `input_tokens + output_tokens` in
`v1.6.5-results.json`.

## How to run

1. Checkout the version under test (`git checkout v1.6.4` or `v1.6.5`).
2. Run `bash activate.sh` (or use the repo as project via `bash install.sh --project .`) so
   steel.md / alloy.md / review-agent files are picked up as the active prompts.
3. Open a fresh Claude Code session in a scratch project with a small TypeScript
   codebase (~500 LOC, any repo with auth + API layer works; we use
   `tests/benchmarks/fixture` if present, else a user-supplied repo).
4. Execute the 12 prompts below, in order, in the same session. Do not use
   `/clear` between turns.
5. After the session ends, record `npx ccusage --json` output.
6. Commit both runs' results to `v1.6.5-results.json`.

## The 12 turns

1. `How is authentication wired in this repo?`
2. `Find all API endpoints that touch the user table.`
3. `What's the data model for sessions?`
4. `Implement a logout endpoint that invalidates the session and clears the cookie.`
5. `Check the new endpoint for security issues.`
6. `Check the new endpoint for performance issues.`
7. `Are there tests for auth flows?`
8. `Add a test for the logout flow — happy path plus expired-token case.`
9. `What dependencies does the auth module pull in?`
10. `Run the tests.`
11. `Any outdated packages in package.json?`
12. `Summarize what was changed and why.`

## What this exercises

- Turn 1–3 trigger RESEARCH path (mercury/graphene)
- Turn 4 triggers BUILD path (tungsten) — 3+ files likely → carbon planner also fires
- Turn 5 triggers SECURITY path (sentinel)
- Turn 6 triggers PERFORMANCE path (iridium)
- Turn 7–8 trigger TESTING path (flint)
- Turn 9, 11 trigger DEPENDENCY path (cobalt)
- Turn 10 is a plain tool invocation
- Turn 12 is a plain summary

Every resident agent prompt (steel, /alloy, review agents) is loaded into at
least one of these turns, so the measured delta reflects realistic cumulative
savings, not cherry-picked best-case paths.

## What counts as a valid measurement

- Same model (e.g. Opus 4.7) on both v1.6.4 and v1.6.5
- Same scratch project, same transcript length (±10%)
- No MCP servers enabled that weren't on in the baseline
- `ccusage` captured at session end, not mid-session

Record honest numbers. If the two runs differ by less than the expected ~15% the
slimming predicts, say so — do not round up.
